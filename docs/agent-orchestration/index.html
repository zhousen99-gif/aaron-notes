04缺口分析 — 需要新增的平台能力CapabilityGap Analysis — Required New Platform Capabilities

按优先级排列，P0 为阻塞项（没有就无None法运转），P1 为重要增强，P2 为体验优化。Prioritized: P0 = blockers (system cannot function without them), P1 = important enhancements, P2 = experience improvements.

P0Agent 间通信Inter-Agent Comm协议Inter-Agent Communication Protocol

需要 send_task_to_agent(agent_name, message) 接口 — L2 在目标 L1 创建新 session 并发送 prompt。这是整个编排架构的基础通道，没有它 L2 就无None法指挥 L1。Requires send_task_to_agent(agent_name, message) API — L2 creates a new session on target L1 and sends a prompt. This is the foundational channel of the entire orchestration architecture; without it, L2 cannot direct L1.

P0任务状态机 / 共享存储Task State Machine / Shared Storage

需要一个 L1 和 L2 都能读写的 任务状态对象（task_id, status, result…），支持 pending → in_progress → completed/failed/escalated 流转。否则 L2 无None法获知 L1 的工作进展。Requires a task state object readable/writable by both L1 and L2 (task_id, status, result...), supporting pending → in_progress → completed/failed/escalated transitions. Without it, L2 cannot know L1's work progress.

P0Session 读取能力CapabilitySession Read Capability

L2 需要 read_agent_session(agent, session_id) 来查看 L1 的执行过程和输出。这是监控和验收的前提。L2 needs read_agent_session(agent, session_id) to view L1's execution process and output. This is a prerequisite for monitoring and validation.

P1Agent 删除 APIAgent Delete API

需要 delete_agent(agent_name) — 对于一次性任务One-time task创建的临时 L1 Agent，完成后应当能清理删除，避免 agent 列表无None限膨胀。Requires delete_agent(agent_name) — temporary L1 Agents created for one-time tasks should be cleanable after completion, preventing unbounded agent list growth.

P1Session 干预能力CapabilitySession Intervention Capability

L2 需要能向 L1 正在执行的 session 注入新指令，或者 L1 支持执行中途的 checkpoint 机制，到 checkpoint 时检查 L2 是否有新指示。L2 needs to inject new instructions into L1's active session, or L1 should support mid-execution checkpoints where it checks for new L2 directives.

P1回调 / 事件通知Callback / Event Notification

L1 完成任务后能 自动触发通知 给 L2（而非 L2 轮询）。类似 webhook 或事件订阅机制。After L1 completes a task, it should automatically trigger notifications to L2 (instead of L2 polling). Similar to webhook or event subscription mechanisms.

P2Agent 能力Capability注册表Agent Capability Registry

每个 Agent 除了 system prompt，增加结构化的 capabilities / domain tags。L2 可通过 search_agents_by_capability(tags) 快速匹配合适的 L1。Beyond system prompts, each Agent would have structured capabilities / domain tags. L2 can quickly match suitable L1s via search_agents_by_capability(tags).

P2任务执行历史Task Execution History

记录每个 Agent 的 历史任务及成功率，帮助 L2 做更精准的任务分配决策。Record each Agent's historical tasks and success rates, helping L2 make more precise task assignment decisions.

05当前可行Feasible的 Workaround 方案Current Feasible Workaround

在平台尚未提供 Agent 间通信Inter-Agent Comm的情况下，可以利用 文件系统 + Schedule 定时任务 搭建一个"穷人版"编排原型。Before the platform provides inter-Agent communication, we can use the file system + scheduled tasks to build a "poor man's" orchestration prototype.

📁 文件系统作为通信总线📁 File System as Communication Bus

所有 Agent 共享一个 workspace 目录，通过 JSON 文件进行异步通信：All Agents share a workspace directory, communicating asynchronously via JSON files:

shared_workspace/ ├── tasks/ │ ├── task_001.json # L2 写入任务定义 (what, who, deadline)# L2 writes task definition (what, who, deadline) │ ├── task_001_status.json # L1 更新执行状态 (progress, blockers)# L1 updates execution status (progress, blockers) │ └── task_001_result.md # L1 写入最终交付物# L1 writes final deliverable ├── escalations/ │ └── esc_001.json # L1 写入求助请求 (问题描述Description, 需要什么)# L1 writes escalation request (issue, what's needed) ├── registry/ │ └── agents.json # Agent 能力Capability注册表Agent Capability Registry (手动维护)# Agent capability registry (manually maintained) └── reports/ └── daily_report.md # L2 生成的汇总报告# Summary report generated by L2

⏱️ Schedule 定时任务作为触发器⏱️ Scheduled Tasks as Triggers

  • L2 发任务：写入 task JSON → 用 create_schedule 创建立即执行的任务，让 L1 去读取并执行L2 assigns tasks: Write task JSON → use create_schedule to create an immediate task, letting L1 read and execute
  • L1 汇报Report：把状态/结果写入文件 → L2 通过定时 schedule 轮询检查L1 reports: Write status/results to files → L2 polls via scheduled tasks
  • L2 干预：写入新的指令文件 → L1 在 prompt 中被要求定期检查指令文件L2 intervenes: Write new instruction files → L1 is prompted to periodically check instruction files
  • L2 汇报Report：定时 schedule 触发 L2 汇总所有 task 状态，生成报告L2 reports: Scheduled task triggers L2 to summarize all task statuses and generate reports
⚠️ Workaround 的局限性：延迟高（非实时）、依赖 Agent 自觉检查文件（无None真正事件驱动）、状态一致性难保证、调试困难。适合用来验证概念和流程，不适合生产级使用。⚠️ Workaround limitations: High latency (not real-time), relies on Agents proactively checking files (no true event-driven), hard to guarantee state consistency, difficult to debug. Suitable for validating concepts and workflows, not for production use.
