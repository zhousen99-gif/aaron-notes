Multi-Level Agent Orchestration

PM Studio 多层级 Agent 编排架构 · 需求与可行性分析
Version 0.1 Draft · 2026-03-31

01愿景概述

构建一个两级 Agent 编排体系：用户只需与少量 L2 编排 Agent 交互，由 L2 自动拆解任务、调度或创建 L1 执行 Agent、监控进度、处理异常，并定期向用户汇报。

核心理念：用户不再直接管理大量 Agent，而是通过 L2 "经理" 来管理 L1 "员工"，实现 任务驱动的自动化编排。

02核心需求拆解

将整个编排架构拆解为 7 项关键能力，每一项都是系统运转的必要条件。

任务生命周期

七项关键能力

# 能力 描述 角色
C1 Agent 生命周期管理 L2 能创建、配置、更新、删除 L1 Agent，包括设计 system prompt、选择 model、配置 skills 和 tools L2
C2 Agent 发现与适配评估 L2 收到任务后，能列举已有 L1 Agent，读取其配置，判断是否适合当前任务 L2
C3 任务下发 L2 能向指定 L1 Agent 发送任务指令，创建新的执行 session L2 → L1
C4 状态回报与求助 L1 能向 L2 汇报任务进度/结果，遇到权限问题或功能缺失时能上报请求帮助 L1 → L2
C5 进度监控与干预 L2 能实时查看 L1 的执行状态，必要时注入新指令或中止任务 L2
C6 定期汇报 L2 能按设定频率向用户汇总所有 L1 的任务状态，生成进度报告 L2 → User
C7 工具使用与执行 L1 能使用内置工具、MCP Servers、编程工具等完成实际任务 L1

03PM Studio 平台现状评估

逐项对照七大能力，评估 PM Studio 当前版本的支持程度。

# 能力 状态 现有支持 缺口说明
C1 Agent 生命周期管理 ⚠ 部分 create_agent_from_config、update_agent 可用 无 delete_agent API，无法"用完删除"
C2 Agent 发现与适配 ✓ 可行 list_agents + 读 profile.json 获取配置 缺少结构化能力标签 (capability tags)，需解析 prompt 推断
C3 任务下发 (L2→L1) ✗ 缺失 仅 create_schedule 可定时触发 无实时 agent-to-agent 消息传递，无法按需即时分配任务
C4 状态回报 (L1→L2) ✗ 缺失 无 无 callback/事件机制，L1 无法主动通知 L2
C5 进度监控与干预 ✗ 缺失 无 L2 无法读取 L1 session 内容，无法注入指令
C6 定期汇报 ✓ 可行 create_schedule 定时触发 Agent 生成报告 汇报质量取决于能否获取 L1 状态 (依赖 C4)
C7 工具使用与执行 ✓ 可行 内置工具 + MCP Servers + 外部集成 已满足需求

04缺口分析 — 需要新增的平台能力

按优先级排列，P0 为阻塞项（没有就无法运转），P1 为重要增强，P2 为体验优化。

P0Agent 间通信协议

需要 send_task_to_agent(agent_name, message) 接口 — L2 在目标 L1 创建新 session 并发送 prompt。这是整个编排架构的基础通道，没有它 L2 就无法指挥 L1。

P0任务状态机 / 共享存储

需要一个 L1 和 L2 都能读写的 任务状态对象（task_id, status, result…），支持 pending → in_progress → completed/failed/escalated 流转。否则 L2 无法获知 L1 的工作进展。

P0Session 读取能力

L2 需要 read_agent_session(agent, session_id) 来查看 L1 的执行过程和输出。这是监控和验收的前提。

P1Agent 删除 API

需要 delete_agent(agent_name) — 对于一次性任务创建的临时 L1 Agent，完成后应当能清理删除，避免 agent 列表无限膨胀。

P1Session 干预能力

L2 需要能向 L1 正在执行的 session 注入新指令，或者 L1 支持执行中途的 checkpoint 机制，到 checkpoint 时检查 L2 是否有新指示。

P1回调 / 事件通知

L1 完成任务后能 自动触发通知 给 L2（而非 L2 轮询）。类似 webhook 或事件订阅机制。

P2Agent 能力注册表

每个 Agent 除了 system prompt，增加结构化的 capabilities / domain tags。L2 可通过 search_agents_by_capability(tags) 快速匹配合适的 L1。

P2任务执行历史

记录每个 Agent 的 历史任务及成功率，帮助 L2 做更精准的任务分配决策。

05当前可行的 Workaround 方案

在平台尚未提供 Agent 间通信的情况下，可以利用 文件系统 + Schedule 定时任务 搭建一个"穷人版"编排原型。

📁 文件系统作为通信总线

所有 Agent 共享一个 workspace 目录，通过 JSON 文件进行异步通信：

shared_workspace/ ├── tasks/ │ ├── task_001.json # L2 写入任务定义 (what, who, deadline) │ ├── task_001_status.json # L1 更新执行状态 (progress, blockers) │ └── task_001_result.md # L1 写入最终交付物 ├── escalations/ │ └── esc_001.json # L1 写入求助请求 (问题描述, 需要什么) ├── registry/ │ └── agents.json # Agent 能力注册表 (手动维护) └── reports/ └── daily_report.md # L2 生成的汇总报告

⏱️ Schedule 定时任务作为触发器

  • L2 发任务：写入 task JSON → 用 create_schedule 创建立即执行的任务，让 L1 去读取并执行
  • L1 汇报：把状态/结果写入文件 → L2 通过定时 schedule 轮询检查
  • L2 干预：写入新的指令文件 → L1 在 prompt 中被要求定期检查指令文件
  • L2 汇报：定时 schedule 触发 L2 汇总所有 task 状态，生成报告
⚠️ Workaround 的局限性：延迟高（非实时）、依赖 Agent 自觉检查文件（无真正事件驱动）、状态一致性难保证、调试困难。适合用来验证概念和流程，不适合生产级使用。

06建议路线图

Phase 1：概念验证 (PoC)现在可以做

基于文件系统 + Schedule 机制搭建原型。验证 L2→L1 任务分发、结果收集、异常上报的核心流程是否符合预期。明确真实使用中的痛点和优先级。

Phase 2：平台能力建设需要 PM Studio 团队支持

推动 P0 能力的平台级实现：Agent 间通信协议、任务状态机、Session 读取 API。这些是从原型走向可靠系统的必要基础设施。

Phase 3：增强与规模化平台能力就绪后

实现 P1/P2 能力：Agent 删除、Session 干预、回调通知、能力注册表。支持更复杂的多 L1 并行任务编排，引入任务执行历史辅助智能调度。

Phase 4：智能化长期目标

L2 具备自主学习能力 — 根据历史数据自动优化 L1 配置、任务分配策略和干预时机。形成真正的 Agentic Workflow Engine。
