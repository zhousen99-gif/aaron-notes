01 Architecture

整体三进程架构

Electron 标准安全模型的最佳实践——渲染进程无法直接访问 Node.js API，所有跨进程调用经过白名单过滤。

整个项目遵循 Electron 标准的多进程安全模型，三个进程分工明确：

  • 渲染进程（Chromium 沙箱）：纯 React 18 UI，只通过 window.electronAPI 与外部通信，无法直接访问 Node.js
  • 预加载进程（安全桥梁）：2,262 行桥接代码，通过 contextBridge.exposeInMainWorld 暴露受控 API，provideInvokeForPreload() 实现白名单过滤——非白名单 channel 直接 throw Error
  • 主进程（Node.js 全能力）：承载所有核心逻辑——AgentChat 对话引擎（4,020 行）、MCPClientManager 工具路由（1,574 行）、AuthManager 认证（1,091 行）、ProfileCacheManager 数据持久化（2,757 行）
💡 设计精髓：Preload 是安全架构的核心中枢。TypeScript 泛型在编译期就保证了白名单的完整性——遗漏任何 key 都会编译报错。
02 IPC Framework

类型安全 IPC 框架（最精妙的设计）

一套类型定义，三个进程共享——TypeScript 泛型 + ES6 Proxy 实现零冗余、编译时安全的跨进程通信。

这是项目最值得关注的工程亮点——用 TypeScript 泛型 + ES6 Proxy 实现了完全类型安全的跨进程通信。一套类型定义，三个进程共享：

角色绑定方式作用
主进程bindMain(ipcMain)Proxy 自动注册 handler，TS 约束回调签名
预加载provideInvokeForPreload(ipc, keys)白名单过滤，编译时检查是否遗漏 key
渲染进程bindRender(invoke)Proxy 拦截调用，自动类型推断
💡 对比传统方式：传统 Electron IPC 手写 channel 字符串容易拼错、参数类型靠人肉检查、新增 API 需改 3 处。OpenKosmos 做到了一处定义、编译器自动推断、零样板代码。
03 Chat Engine

Agent 对话引擎——核心循环与工具调用

agentChat.ts（4,020 行）——整个系统的心脏。用户消息 → 上下文管理 → 压缩检查 → LLM 调用 → 工具循环。

从用户按下发送到收到回复，经历完整的对话主循环：

  1. 保存用户消息到上下文（AddMessageToSession）
  2. 进入 while 循环，每轮迭代：
    • 检查 CancellationToken（用户是否点了停止）
    • CheckAndCompress：token 超限则自动压缩上下文
    • callWithToolsStreaming：调用 LLM API，流式推送 token
    • 判断响应类型：纯文本→结束循环；工具调用→进入工具分支
  3. 工具分支：批量审批 → 逐个执行 → 后处理 → 结果回填上下文 → 回到循环顶部
💡 关键设计：批量审批而非逐个弹窗，减少用户交互打断；图片注入自动从 MCP 工具结果中提取 base64 并压缩后作为 user image message 注入，供视觉模型分析。
04 MCP Routing

MCP 工具路由机制

MCPClientManager（1,574 行）实现了工具名到服务器的透明路由——AgentChat 只需调用 executeTool(name, args)，无需感知工具在哪。

  • 内置工具（read_file、web_search 等 30+ 个）：通过 BuiltinMcpClient 直接在主进程执行
  • 外部 MCP 工具：通过 toolToServerMap（Map<toolName, serverName>）查询后路由到对应的 VscMcpClient——支持 stdio/SSE/Streamable HTTP 三种传输

并发安全通过两层机制保障：

  • OperationLock：每个 MCP 服务器独立操作锁，防止 connect/disconnect/reconnect 并发冲突
  • AbortController：追踪进行中的连接，用户切换 Agent 时可立即中止旧连接
05 Compression

上下文压缩机制

fullModeCompressor.ts（645 行）——Token 超限时自动触发，用轻量模型生成结构化摘要替换中间消息，优雅解决长会话问题。

保留策略（绝不压缩的部分）：

  • 🔒 首条用户消息：确立对话目标
  • 🔒 首次技能调用：工具发现记录
  • 🔒 最近 5 条消息：维持连贯性

压缩部分（中间所有消息）被发送给轻量的 claude-haiku 模型，生成包含 8 个固定章节的结构化摘要：对话概述、资源基础、内容状态、问题解决、进度追踪、活跃工作、最近操作、续接计划——确保 AI 能无缝续接，不丢失任何关键上下文。

💡 递归巧妙：用 LLM 来压缩 LLM 的记忆。选用轻量的 claude-haiku 而非主力模型，压缩本身的 token 消耗极低。
06 Cancellation

取消令牌（Cancellation Token）模式

借鉴 VSCode 的 CancellationToken 模式，实现 Source/Token 职责分离——拥有者控制取消、执行者只读检查。

  • CancellationTokenSource（拥有者）：持有 cancel() 和 dispose() 方法
  • CancellationToken（只读接口）：执行者只能检查 isCancellationRequested，不能触发取消
  • AgentChatManager 为每个 chatSession 持有一个 Source

在对话循环中设置了 7 个取消检查点：streamMessage 入口、while 循环开始、压缩完成后、工具审批前、工具审批后、每个工具执行前、错误捕获层——每个异步阻塞节点前后都检查，保证用户点停止后毫秒级生效。

07 Persistence

数据持久化分层架构

ProfileCacheManager（2,757 行）是数据中枢，实现了内存缓存 → 磁盘持久化 → IPC 同步的三层分离。

层次职责实现
内存层快速读写Map<alias, ProfileV2> / Map<sessionId, AgentChat> / Map<serverName, RuntimeState>
磁盘层跨会话持久化profile.json（Agent 配置）/ auth.json（OAuth token）/ session.json（对话历史）
IPC 同步层前端实时刷新profile:cacheUpdated / mcp:statusChanged / agentChat:chatStatusChanged
💡 核心原则：缓存是加速层而非替代品。每次修改后同时写磁盘 + 推 IPC，保证三层永远一致。前端不需要轮询，通过 IPC 事件驱动 React 状态自动更新。
08 Multi-Agent

多 Agent 并发管理

AgentChatManager（1,422 行）以 ChatSessionId 为粒度管理独立的 AgentChat 实例，5 分钟空闲自动回收。

实例生命周期 5 个阶段：

  1. 创建：用户打开 Tab → new AgentChat(alias, chatId, sessionId)，若有历史则从 session.json 恢复
  2. 活跃：用户发消息 → streamMessage()，每次交互重置 idle 计时器
  3. 空闲：用户切换到其他 Tab → 启动 5 分钟倒计时
  4. 销毁：5 分钟无操作 → destroy()，释放内存 + 清理 CancellationSource
  5. 重建：用户切回已销毁 Tab → 从 session.json 重新创建实例，用户无感知恢复
💡 设计要点：实例池 + 惰性清理，避免多标签页同时开启时的内存泄漏，同时保证对话上下文不丢失。
