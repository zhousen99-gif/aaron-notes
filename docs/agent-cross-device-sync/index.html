当 AI Agent 需要搬家When Your AI Agent Needs to Move

跨设备同步的真实代价，以及"把文件变小"带来的启发The real cost of cross-device sync, and what "making files smaller" taught me
2026-04-09 · Aaron Zhou

起因The Setup

我有两台 Windows 电脑，上面各跑着一套 PM Studio——一个基于 Electron 的 AI Agent 桌面平台。主力机上有 10 个 Agent、14 个 Skill、大量对话历史和 Knowledge Base；另一台只有 2 个默认 Agent。PM Studio 目前不支持多设备同步，所以我决定自己造一个。I have two Windows machines, each running PM Studio — an Electron-based AI Agent desktop platform. The primary has 10 Agents, 14 Skills, extensive chat history and Knowledge Bases; the other had just 2 default Agents. PM Studio doesn't support multi-device sync, so I decided to build one myself.

第一版方案很简单：把所有配置文件打包，通过 OneDrive 传到另一台机器，解压覆盖。全量包 274 MB，传一次需要几分钟。跑通了。The first version was simple: package all config files, transfer via OneDrive, extract and overwrite. Full package: 274 MB, a few minutes to transfer. It worked.

但用了两天之后我意识到，日常同步的需求其实和初次迁移完全不同。每次都传 274 MB 是一种巨大的浪费。But after two days of use, I realized daily sync needs are fundamentally different from initial migration. Transferring 274 MB every time is massive waste.

274 MB 里到底装了什么What's Inside Those 274 MB

把全量包拆开看，数据分布极不均匀：Breaking down the full package reveals a highly uneven distribution:

组件Component大小Size变化频率Change Frequency说明Notes
profile.json156 KB频繁FrequentAgent 配置、model、skills 分配Agent config, model, skills assignment
app.json0.7 KB极少RareUI 偏好UI preferences
skills/40 MB很少Rare安装/更新 skill 时才变Only changes on skill install/update
knowledge bases251 MB很少Rare给 Agent 添加文档时才变Only changes when adding docs to Agent
chat_sessions/47 MB每次对话Every conversation新对话持续追加New sessions constantly appended

274 MB 中有 251 MB 是 knowledge base，而它几乎不变。 每次全量同步都在重复传输一堆 PDF、PPT 和参考文档——这些文件可能几周甚至几个月都不会改动一次。251 MB of the 274 MB is knowledge base, and it barely changes. Every full sync re-transfers a bunch of PDFs, PPTs, and reference docs — files that may not change for weeks or months.

真正频繁变化的只有两样东西：profile.json（改了 Agent 配置就变）和 chat_sessions（每次对话都变）。它们加起来不到 50 MB，压缩后只有 18 MB。What actually changes frequently is just two things: profile.json (changes with any Agent config edit) and chat_sessions (changes with every conversation). Together they're under 50 MB, compressed to just 18 MB.

这意味着，每天 93% 的传输量都是在搬运不会变的数据。This means 93% of daily transfer volume is moving data that hasn't changed.

按变化频率切分，而不是按文件类型Split by Change Frequency, Not File Type

第一个直觉是做增量同步——比较文件哈希，只传有变化的文件。但这会引入大量复杂度：要在两台机器之间维护一个文件清单、计算差异、处理冲突。对于一个个人工具来说，这不值得。The first instinct was incremental sync — compare file hashes, transfer only changes. But this introduces massive complexity: maintaining a file manifest across two machines, computing diffs, handling conflicts. For a personal tool, it's not worth it.

更好的思路是：不追踪什么变了，而是接受"有些东西几乎不变"这个事实，按变化频率分层。A better approach: instead of tracking what changed, accept that "some things rarely change" and tier by change frequency.

最终设计是两个模式：The final design has two modes:

模式Mode内容Content大小Size场景When to Use
快速同步Quickprofile.json + app.json + chat_sessions~18 MB日常：改了配置或聊了天Daily: config changes or new chats
全量同步Full上面 + skills + knowledge basesAbove + skills + knowledge bases~274 MB偶尔：装了新 skill 或加了知识库Occasional: new skills or KB files

这个分层不复杂——两个 .cmd 文件，双击就能运行。但它把日常同步从 274 MB 降到了 18 MB，从"等几分钟"变成"几秒钟搞定"。Nothing complex — two .cmd files, double-click to run. But it cuts daily sync from 274 MB to 18 MB, from "wait a few minutes" to "done in seconds."

这个设计有一个隐含假设：导入端如果没有 skills.tar.gz，就跳过 skills 步骤，保留本地现有的。 这要求两台机器的 skills 基线是一致的——第一次必须做全量同步，之后快速模式才能工作。这和 Git 的工作方式类似：你需要一次完整的 clone，之后才能用增量的 pull。This design has an implicit assumption: if the import side doesn't find skills.tar.gz, it skips skills and keeps the local copy. This requires both machines to have a consistent skills baseline — the first sync must be full, then quick mode works. Similar to Git: you need a full clone before incremental pulls work.

一个更深的启发：Purview DLP 对同步方案的约束A Deeper Insight: How Purview DLP Constrains Sync Design

分级同步解决了"文件太大"的问题，但还有一个维度的约束来自企业安全策略。在用 OneDrive 作为传输通道时，Microsoft Purview 的数据泄露防护（DLP）会扫描所有文本文件，检测 API Key、Client Secret、Access Token 等敏感信息模式。一旦触发，文件会被标记为 "sensitive data blocked"，可能被锁定甚至隔离。Tiered sync solved the "files too large" problem, but another constraint came from enterprise security policy. When using OneDrive as the transfer channel, Microsoft Purview's Data Loss Prevention (DLP) scans all text files, detecting API Keys, Client Secrets, Access Tokens, and other sensitive patterns. Once triggered, files get flagged as "sensitive data blocked" — potentially locked or quarantined.

这不是 bug，是 feature。Purview 做的是对的——profile.json 里确实存着明文的 API Secret，放在 OneDrive 上确实不应该。This isn't a bug, it's a feature. Purview is right — profile.json does contain plaintext API Secrets, and they shouldn't sit in OneDrive unprotected.

我的应对方式有两层：My response has two layers:

  • profile.json 中的敏感字段用 Base64 混淆（OBF:base64 前缀），导入时自动解码还原。这不是加密——只是让 DLP 的模式匹配识别不到明文 API Key 格式。Sensitive fields in profile.json are obfuscated with Base64 (OBF:base64 prefix), auto-decoded on import. This isn't encryption — it just prevents DLP's pattern matching from recognizing plaintext API Key formats.
  • skills、knowledge base、chat_sessions 全部用 tar.gz 压缩。 Purview 不扫描压缩包内部文件内容。Skills, knowledge bases, and chat_sessions are all packed as tar.gz. Purview doesn't scan file contents inside compressed archives.

这件事让我意识到，企业环境下的同步方案必须把安全策略当成一等约束来设计，而不是事后打补丁。 DLP 的存在意味着你不能假设"文件放上去就能被另一台设备正常读取"。传输通道上有看不见的审查者。这和分布式系统里的另一条经典原则呼应：不要假设网络是安全的——延伸到企业语境，就是不要假设存储通道是透明的。This made me realize that sync solutions in enterprise environments must treat security policies as first-class design constraints, not afterthought patches. DLP's existence means you can't assume "upload a file and the other device reads it fine." There are invisible auditors on the transfer channel. This echoes another classic distributed systems principle: don't assume the network is secure — extended to enterprise context, don't assume the storage channel is transparent.

同步的尽头不是技术，是产品决策At the End of Sync, It's a Product Decision

回头看这整个过程，我越来越觉得"跨设备同步"这件事本身就是一个信号——它暴露了当前 AI Agent 平台在架构上的一个根本缺陷：状态绑定在本地文件系统。Looking back at this entire process, I increasingly feel that "cross-device sync" itself is a signal — it exposes a fundamental architectural gap in current AI Agent platforms: state is bound to the local filesystem.

PM Studio 把 Agent 的配置（profile.json）、技能（skills/）、记忆（knowledge base）、对话历史（chat_sessions/）全部存在 AppData/Roaming 下。这是 Electron 桌面应用的标准做法，但一旦用户需要在多台设备间切换，所有分布式系统的经典难题就全部涌上来：状态同步、冲突解决、数据一致性、安全策略兼容。PM Studio stores Agent config (profile.json), skills (skills/), memory (knowledge base), and chat history (chat_sessions/) all under AppData/Roaming. Standard practice for Electron desktop apps, but the moment users need multi-device usage, all classic distributed systems challenges emerge: state synchronization, conflict resolution, data consistency, security policy compatibility.

这不只是 PM Studio 的问题。今天几乎所有桌面 AI Agent 平台——Claude Desktop、Cursor、Windsurf——都面临相同的架构约束。它们都是"单机应用"的后代，继承了本地优先（local-first）的存储哲学。This isn't just PM Studio's problem. Nearly all desktop AI Agent platforms today — Claude Desktop, Cursor, Windsurf — face the same architectural constraint. They're descendants of "single-machine apps," inheriting a local-first storage philosophy.

Martin Fowler 在讨论事件驱动架构时区分了 orchestration 和 choreography。我最初想用 OneDrive symlink 做 choreography——让两台设备通过共享文件自行协调，没有中心控制。结果不行：文件冲突、DLP 拦截、并发写入，系统越复杂越难调试。最终我选了 orchestration——显式的导出/导入，有一个明确的控制流程。在两台设备之间同步配置时，"清晰"比"自动"更有价值。Martin Fowler, discussing event-driven architecture, distinguished orchestration from choreography. I initially tried choreography via OneDrive symlinks — let two devices self-coordinate through shared files, no central control. It failed: file conflicts, DLP blocking, concurrent writes — the more complex the system, the harder to debug. I chose orchestration — explicit export/import, a clear control flow. When syncing config between two devices, "clarity" is more valuable than "automation."

我越来越相信，未来真正好的 Agent 平台会把"配置即服务"做成基础设施——profile、skills、memory 存在云端，本地只是渲染终端。就像 VS Code 的 Settings Sync 那样，但覆盖范围更深，涵盖 Agent 的整个上下文层。在那之前，分级导出/导入大概是最务实的权宜方案。I increasingly believe that truly great Agent platforms will eventually make "config-as-a-service" part of their infrastructure — profile, skills, memory stored in the cloud, local device as just a rendering terminal. Like VS Code's Settings Sync, but deeper, covering the Agent's entire context layer. Until then, tiered export/import is probably the most pragmatic stopgap.

不过话说回来，这件事也让我重新审视了一个朴素的工程直觉：你传输的数据量，应该和信息变化量成正比，而不是和信息总量成正比。 274 MB 降到 18 MB，不是因为压缩算法更好了，而是因为我终于搞清楚了"什么在变，什么没变"。这个道理不新——数据库的 WAL（Write-Ahead Log）、Git 的 delta、CDN 的缓存分层，本质上都在做同一件事：只传输变化。That said, this experience also reinforced a fundamental engineering intuition: the amount of data you transfer should be proportional to the amount of change, not the total amount of data. 274 MB down to 18 MB — not because of better compression, but because I finally understood "what changes vs. what doesn't." This isn't new — database WAL (Write-Ahead Log), Git's delta encoding, CDN cache tiering all do the same thing: only transfer the delta.

区别在于，这些成熟系统有精确的变更追踪机制。而我的方案更粗暴——不追踪变更，直接按"变化频率"分层。精确度不如 delta sync，但复杂度也低得多。对于一个两台设备之间的个人工具来说，这就够了。The difference is that those mature systems have precise change-tracking mechanisms. My approach is cruder — no change tracking, just tiering by "change frequency." Less precise than delta sync, but far less complex. For a personal tool between two devices, it's enough.

有时候，最好的优化不是让算法更聪明，而是让自己对数据更诚实。Sometimes the best optimization isn't a smarter algorithm — it's being more honest about your data.
